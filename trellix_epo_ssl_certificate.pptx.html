OVERVIEW

Why Self-Signed Certificates?

Internal Networks & Lab Environments

Self-signed certificates are ideal for internal infrastructure where you control the trust environment. Perfect for testing, development, and isolated production networks.

When You Cannot Wait for a CA

Certificate Authority (CA) validation can take days or weeks. Self-signed certificates provide immediate encryption for urgent deployments or time-sensitive projects.

Basic Security & Encryption

Provides end-to-end encryption for data in transit. While not automatically trusted by browsers, the cryptographic protection is identical to CA-signed certificates.

What Makes This Method Secure?

Password-Protected Private Key

Industry best practice for key security

2048-bit RSA Encryption

Trellix recommended minimum

Subject Alternative Name (SAN)

Required by modern browsers

ePO 5.10.0+ Compatible

Works with latest OpenSSL versions

Important Note

This guide follows Trellix official documentation and industry best practices. The method is specifically designed for ePolicy Orchestrator 5.10.0 and newer versions.

STEP 1

Download and Install OpenSSL

What is OpenSSL?

OpenSSL is a free, open-source toolkit for creating SSL/TLS certificates. Trellix recommends using OpenSSL for certificate generation due to its reliability and industry-wide adoption.

Used by 70%+ of web servers globally

Download Location

https://slproweb.com/products/Win32OpenSSL.html

Why "Light" Version?

The Light version contains all essential tools needed for certificate creation. The full version includes extra development files not required for this task.

STEP 3

Create OpenSSL Configuration File

Why Configuration File?

Modern browsers (Chrome, Edge, Firefox) require Subject Alternative Name (SAN) in certificates. Without SAN, browsers show error:

NET::ERR_CERT_COMMON_NAME_INVALID

"does not specify Subject Alternative Names"

The configuration file adds SAN to your certificate automatically.

Create the File

1

Open Notepad (Run as Administrator)

2

Copy and paste the configuration (see right panel)

3

Modify the values for your environment

4

Save as: C:\ssl\openssl.cnf

Important Notes

  • • CN (Common Name) must match your ePO server IP or hostname
  • • IP.1 in alt_names enables IP-based access
  • • If using hostname, add DNS.1 = your-hostname

Configuration File Content 

[req]
default_bits = 2048
prompt = no
default_md = sha256
distinguished_name = dn
x509_extensions = v3_req

[dn]
C = IN
ST = Maharashtra
L = Mumbai
O = CyberSecLab
OU = IT
CN = 192.168.0.100

[v3_req]
subjectAltName = @alt_names

[alt_names]
IP.1 = 192.168.0.100

Field Descriptions

Field Description Example
C Country code (2 letters) IN, US, UK
ST State or Province Maharashtra
L City/Locality Mumbai
O Organization name CyberSecLab
OU Department/Unit IT
CN Server IP or hostname 192.168.0.100
IP.1 Server IP address 192.168.0.100

Generating secure keys and certificates with OpenSSL

15 min process

STEP 4

Generate Password-Protected Private Key

Why Password Protection?

Password-protecting the private key adds an extra layer of security. Even if someone gets access to the key file, they cannot use it without the passphrase. This is an industry best practice.

Run This Command

"C:\Program Files\OpenSSL-Win64\bin\openssl.exe" genrsa -des3 -out C:\ssl\keys\ca.key 2048

What You'll See:

Enter PEM pass phrase: ********

Verifying - Enter PEM pass phrase: ********

Important

  • • Remember your passphrase! You need it in the next step.
  • • Use a strong passphrase (mix of letters, numbers, symbols)
  • • The passphrase is NOT displayed while typing (security feature)

Command Explanation

genrsa

Generate RSA private key

-des3

Encrypt with Triple DES (password protection)

-out

Output file location

2048

Key size in bits (Trellix recommended minimum)

Output

File created:

C:\ssl\keys\ca.key

This is your password-protected private key. Keep it secure!

STEP 5

Create Self-Signed Certificate with SAN

What is a Self-Signed Certificate?

A certificate where the issuer and subject are the same entity (your organization). It's signed by your own private key instead of a trusted Certificate Authority. Perfect for internal use and testing environments.

Run This Command

"C:\Program Files\OpenSSL-Win64\bin\openssl.exe" req -new -x509 -days 365 -key C:\ssl\keys\ca.key -out C:\ssl\certs\ca.cer -config C:\ssl\openssl.cnf

You'll Be Prompted For:

Enter pass phrase for C:\ssl\keys\ca.key: ********

Enter the passphrase you created in Step 4.

Output

File created:

C:\ssl\certs\ca.cer

Your self-signed certificate with SAN support

Command Explanation

req

Certificate request command

-new

Create new request

-x509

Output self-signed certificate

-days 365

Valid for 1 year

-key

Use this private key

-out

Output certificate file

-config

Use configuration file (includes SAN)

Validity Period

Certificate is valid for 365 days. Mark your calendar to renew before expiration!

STEP 6

Convert Private Key to RSA Traditional Format

Why This Step?

OpenSSL 3.x creates keys in PKCS#8 format, but ePO requires Traditional RSA format. This conversion ensures ePO can read the private key correctly.

PKCS#8 Format

-----BEGIN PRIVATE KEY-----

RSA Format

-----BEGIN RSA PRIVATE KEY-----

Run This Command

"C:\Program Files\OpenSSL-Win64\bin\openssl.exe" rsa -in C:\ssl\keys\ca.key -out C:\ssl\keys\ca-rsa.key -traditional

You'll See:

Enter pass phrase for C:\ssl\keys\ca.key: ********

writing RSA key

Note

The converted key (ca-rsa.key) is unencrypted for ePO compatibility. Keep this file secure!

Command Explanation

rsa

RSA key processing command

-in

Input encrypted key

-out

Output RSA format key

-traditional

Use traditional RSA format

Output

File created:

C:\ssl\keys\ca-rsa.key

Unencrypted, RSA format for ePO

Key Files Summary

ca.key Encrypted (backup)
ca-rsa.key For ePO (use this)

STEP 7

Verify Certificate (Optional but Recommended)

Why Verify?

Before uploading to ePO, confirm the certificate contains correct information and SAN. This step catches configuration errors early and saves troubleshooting time later.

Run This Command

"C:\Program Files\OpenSSL-Win64\bin\openssl.exe" x509 -in C:\ssl\certs\ca.cer -text -noout

Confirm These Items

Subject shows your organization details

CN matches your server IP

Subject Alternative Name shows your IP address

Validity period is correct (365 days)

What to Look For 

Subject: C = IN, ST = Maharashtra, 
         L = Mumbai, O = CyberSecLab, 
         OU = IT, CN = 192.168.0.100

X509v3 extensions:
    X509v3 Subject Alternative Name:
        IP Address:192.168.0.100

Validity:
    Not Before: Feb 5 00:00:00 2026 GMT
    Not After : Feb 5 00:00:00 2027 GMT

Subject

Verify your organization details are correct

Subject Alternative Name

Critical for modern browser compatibility

Validity Period

Confirm 365-day validity from creation date

STEP 9

Restart ePO Server

Why Restart?

ePO needs to reload with the new certificate. The change won't take effect until restart. This is a critical step - skipping it will result in the old certificate remaining active.

1

Command Prompt

Recommended method

net stop "McAfee ePolicy Orchestrator Application Server"

Wait for service to stop, then:

net start "McAfee ePolicy Orchestrator Application Server"

2

Services Console

GUI alternative

1

Press Win + R

2

Type services.msc → Press Enter

3

Find "McAfee ePolicy Orchestrator Application Server"

4

Right-click → Restart

3

Restart Server

Nuclear option

If other methods don't work, restart the entire Windows server.

Warning: This will interrupt all services on the server.

Service Restart Time

The ePO Application Server typically takes 2-5 minutes to fully restart. Be patient and wait for the service to show "Running" status.

STEP 10

Import Certificate to Browser (Remove Warning)

Why Import?

Self-signed certificates are not trusted by browsers by default. Importing to the Trusted Root store removes the "Not secure" warning and provides a seamless user experience.

Steps for Windows

1

Navigate to C:\ssl\certs\

2

Double-click on ca.cer

3

Click Install Certificate

4

Select Local Machine → Click Next

5

Select "Place all certificates in the following store"

6

Click Browse → Select "Trusted Root Certification Authorities"

7

Click OK → Next → Finish

8

Click Yes to confirm the security warning

Restart Browser

1

Close ALL browser windows completely

2

Reopen browser

3

Navigate to https://192.168.0.100:8443

Expected Result

Certificate shows your organization name (CyberSecLab)

Certificate shows correct validity dates

Connection is encrypted

No more "Not secure" warning

Success!

Your Trellix ePO server is now secured with a self-signed SSL certificate!

TROUBLESHOOTING

Troubleshooting Common Errors

"Private key file did not contain a valid private key"

Cause:

Using wrong key file or key format

Solution:

Use ca-rsa.key (converted RSA format), not ca.key

"NET::ERR_CERT_COMMON_NAME_INVALID"

Cause:

Certificate missing SAN

Solution:

Ensure openssl.cnf has [alt_names] section with correct IP

Browser Still Shows "Not Secure"

Cause:

Certificate not imported to trusted store

Solution:

Follow Step 10 to import certificate to Trusted Root CA

Certificate Shows Old Information After Upload

Cause:

ePO not restarted

Solution:

Restart ePO Application Server service

Cannot Find Services

Service Name:

McAfee ePolicy Orchestrator Application Server

Alternative:

Restart the entire Windows server

BEST PRACTICES

Security Best Practices

Essential recommendations for maintaining certificate security

1

Keep Private Keys Secure

Never share ca.key or ca-rsa.key files. Store them in an encrypted location with restricted access.

2

Use Strong Passphrase

Minimum 12 characters with mixed types (uppercase, lowercase, numbers, symbols). Avoid dictionary words.

3

Limit File Access

Restrict C:\ssl\keys\ folder permissions to administrators only. Use NTFS permissions.

4

Renew Before Expiry

Certificate valid for 365 days. Set calendar reminder 30 days before expiration to avoid service disruption.

5

Backup Files

Keep secure backup of certificate files in an encrypted offline location. Test restoration process periodically.

6

Production Use

Consider using CA-signed certificate for production environments with external access requirements.

Critical Security Reminder

Self-signed certificates provide encryption but do not provide identity verification by default. They are ideal for internal networks and testing. For production environments accessible from the internet, consider using certificates from a trusted Certificate Authority (CA) such as DigiCert, Let's Encrypt, or Sectigo.
